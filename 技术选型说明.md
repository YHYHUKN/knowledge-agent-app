# 技术选型说明

本文档说明本项目各技术层的选型理由与实现细节。

---

## 一、前端框架

| 技术 | 版本 | 说明 |
|------|------|------|
| Next.js | 14.2.5 | App Router 模式，前后端一体化，无需独立服务；服务端渲染对 SEO 友好 |
| React | 18.3.1 | 组件化，生态成熟 |
| TypeScript | 5.5.4 | 严格类型检查，全项目无 any 逃逸 |

---

## 二、UI 组件库

| 技术 | 说明 |
|------|------|
| Ant Design 5 | 企业级 ToB 组件体系，主题定制能力强；覆盖 Button、Card、Modal、Table、Tag、Tooltip、Skeleton 等全部业务组件 |
| Tailwind CSS | 辅助布局与响应式，使用 CSS 变量 token 覆盖 Ant Design 默认色值 |
| Recharts | 轻量级图表库，用于可观测统计折线图 |
| @ant-design/icons + lucide-react | 互补覆盖：antd 图标库满足功能图标，lucide 补充 UI 细粒度图标 |

---

## 三、后端

| 技术 | 说明 |
|------|------|
| Next.js API Route | 与前端共用同一框架，开发/部署零额外成本；无需独立后端服务 |
| globalThis 内存 | Node 进程内存存储，API Route 共享同一进程上下文，无需额外数据库连接 |
| globalThis 内存 + localStorage 双写 | 兼顾服务端检索计算（内存）和用户数据持久化（localStorage） |

---

## 四、持久化方案

采用 **客户端 localStorage + 服务端内存** 的混合方案：

- **服务端内存**（`lib/data.ts`）：API Route 需要在服务端做检索计算，必须持有数据。`globalThis` 挂载避免 Next.js HMR 重置。
- **客户端 localStorage**（`lib/client-storage.ts`）：用户新增的资产通过 `fetch` POST 到 API 后，客户端同步写入 localStorage，刷新页面 / 重启浏览器后数据保留。
- **未使用** SQLite / JSON 文件 / PostgreSQL：笔试场景下无需引入额外依赖，服务端文件系统访问在部分部署环境受限。

---

## 五、检索方案

选用 **方案①：关键词匹配 + 加权打分**（`lib/search.ts`），理由：

- 零外部依赖，纯 Node 内存计算
- 延迟极低，可解释性强
- 与 `SearchResult` 接口契约稳定，上层代码（API / 前端）无需感知检索实现差异
- 升级为向量检索时，只需替换 `lib/search.ts` 内部逻辑，接口不变

---

## 六、代码组织

```
lib/           纯函数业务逻辑，无 React 依赖，可单独做单元测试
app/api/       API Route，HTTP 接口层
components/    React 组件，依赖 Next.js / Ant Design
```

分层清晰，核心检索 / Agent 逻辑与 UI 完全解耦，便于后续升级（如替换为真实 LLM）。
